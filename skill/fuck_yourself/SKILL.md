当用户对产出提出方向性质疑（非细节修订，而是指出思路、原则层面的问题）时，执行自我反省流程。

用户输入: $ARGUMENTS

## 前置判断

在执行反省前，先确认质疑是否属于本 skill 的范围：

| 质疑类型 | 示例 | 处理方式 |
|----------|------|----------|
| **方向性质疑** | "你不应该这样设计"、"思路完全错了" | → 进入反省流程 |
| **细节修订** | "这里变量名改一下"、"缩进不对" | → 直接修改，不触发反省 |
| **偏好表达** | "我更喜欢用 X 而不是 Y" | → 存 feedback memory，不触发反省 |

如果判定为细节修订或偏好表达，告知用户不触发反省并说明原因，然后直接处理。

## 步骤

### 1. 还原现场

回溯本次对话中与质疑相关的完整决策链：

1. **定位出错产出**：找到用户质疑的具体代码/文档/方案
2. **追溯决策点**：从产出往回找，列出导致该产出的每一个关键决策（选了什么方案、为什么选、当时参考了什么信息）
3. **标记偏离点**：在决策链中标出"从哪一步开始偏离了用户期望"

输出格式：
```
决策链：
  1. [决策内容] — 依据: [参考信息]
  2. [决策内容] — 依据: [参考信息]  ← 偏离点
  3. ...
```

### 2. 归因

基于偏离点，识别错误的根本原因。逐项排查以下可能性（按优先级）：

| 优先级 | 原因类型 | 判断标准 | 典型表现 |
|--------|----------|----------|----------|
| 1 | **规则缺失** | 当前 CLAUDE.md 没有覆盖这类场景 | 首次遇到该类决策，无规则可循 |
| 2 | **规则模糊** | 规则存在但有多种解读，选了错误的解读 | 规则措辞不够精确，留了歧义空间 |
| 3 | **规则冲突** | 多条规则同时适用但互相矛盾 | 按 A 规则该这样做，按 B 规则该那样做 |
| 4 | **规则过窄** | 规则存在但只覆盖了部分场景 | 规则在类似场景下有效，但本场景是边界情况 |
| 5 | **执行偏差** | 规则正确且清晰，但执行时未遵守 | 规则已经说了该怎么做，没有照做 |

只选一个最核心的原因类型。如果是"执行偏差"，说明为什么没遵守（上下文过长遗漏？优先级误判？）。

输出：一句话归因 + 原因类型

### 3. 泛化

将修正抽象为通用原则，遵循以下约束：

1. **检索现有规则**：读取项目 CLAUDE.md，找到与本次质疑相关的所有现有规则
2. **决定操作类型**：
   - 规则缺失 → 新增原则
   - 规则模糊/过窄 → 修改现有原则的措辞，扩大覆盖范围或消除歧义
   - 规则冲突 → 合并或重新排序冲突规则，明确优先级
   - 执行偏差 → 在规则中增加强调标记（如 IMPORTANT）或前置条件检查
3. **质量检查**：新/改原则必须同时满足：
   - 不针对单一场景做特化补丁（能覆盖同类未来场景）
   - 不与现有其他规则矛盾
   - 可验证（能判断是否在遵守）
   - 足够简洁（一条规则不超过两句话，除非需要表格辅助）

输出：操作类型 + 修正后的原则文本 + 覆盖的场景范围（用 2-3 个例子说明）

### 4. 向用户确认

将归因结论和泛化后的原则文本展示给用户，请求确认：

- 归因是否准确
- 新原则的措辞和覆盖范围是否合理
- 是否有需要调整的地方

**用户确认后才进入后续写入步骤。** 如果用户提出修改意见，回到步骤 3 调整。

### 5. 存档

更新前将当前项目 CLAUDE.md 备份：

1. 检查 `.claude/claude_history/` 目录是否存在，不存在则创建
2. 读取已有备份文件，确定当前最大版本号
3. 复制到 `.claude/claude_history/CLAUDE.v{主版本}_{次版本}.md`，次版本递增
4. 主版本号在 CLAUDE.md 的角色或项目概览发生重大变更时递增，其余情况只递增次版本
5. 在备份文件末尾追加变更记录，格式如下：

```markdown

---

## 变更记录

- **日期**: {当前日期}
- **触发原因**: {用户质疑的原话摘要}
- **归因**: {步骤 2 的一句话归因}
- **更新思路**: {步骤 3 的操作类型 + 原则变更要点，一句话概括}
```

### 6. 更新

修改项目 CLAUDE.md：

1. **定位插入点**：根据原则的语义，找到 CLAUDE.md 中最合适的 section
2. **执行变更**：
   - 新增原则 → 插入到对应 section，保持该 section 内的逻辑顺序
   - 修改原则 → 原地编辑，不改变位置
   - 合并规则 → 删除被合并的条目，在保留条目处更新
3. **同步根目录**：如果根目录存在 `CLAUDE.md`（与 `.claude/CLAUDE.md` 不同文件），同步相同变更

### 7. 沉淀

将反省结论写入 memory（feedback 类型），包含：
- 规则内容
- **Why:** 触发原因（引用用户的原话）
- **How to apply:** 应用场景（列出 2-3 个具体情况）
